## 0. Background

`cc-trigger` (Node.js systemd service on KVM8 at `/opt/stockbuddy/cc-trigger/server.js`) is the partial "claude-code-manager" that today does:

1. ✅ Receives alert payload at `POST 127.0.0.1:8765/alert`.
2. ✅ Heuristically picks repo (TTE vs Stock Buddy).
3. ✅ Spawns `claude --print` with an "on-call engineer" investigation prompt → Claude opens a PR.
4. ❌ Reviewer Claude that audits Writer Claude's diff. — **TODO**
5. ❌ CI gate (`gh pr checks` green before merge). — **TODO**
6. ❌ Auto-merge + auto-deploy (`git pull`, `docker compose up -d --build`). — **TODO**
7. ❌ 10-min post-deploy watchdog + auto-revert on regression. — **TODO**

Plus a **pre-step-0 gap**: no monitoring checks exist for TTE failure modes (snapshot worker, change_symbol errors, alert sidebar timeouts), so steps 1–7 never fired during the 6-day snapshot blackout that began 2026-05-08 15:13 UTC. cc-trigger is only as useful as the alerts feeding it.

This spec implements steps 4–7 AND adds the missing TTE monitoring checks so the full automation actually fires.

## 4. Component-by-component spec

### 4.1 cc-reviewer (G1)

**Purpose**: For every PR Writer Claude opens, spawn a Reviewer Claude to audit it before merging.

**Trigger**: GitHub webhook (`pull_request` opened/synchronized event) routed to a new local endpoint `POST 127.0.0.1:8766/pr`. Webhook secret stored in `/opt/stockbuddy/secrets/.env.cc`.

**Behavior**:
1. Receive PR payload, ignore PRs not authored by the bot.
2. Spawn `claude --print` with a Reviewer prompt that:
   - Reads the PR diff (`gh pr diff <number>`)
   - Reads the linked issue/alert payload from `/opt/stockbuddy/cc-state/pr-<n>.json`
   - Audits for bugs, missing tests, security issues, unintended scope, convention violations
   - Posts a structured review via `gh pr review <number> --comment|--approve|--request-changes`
   - Sets label `reviewed-pass` or `reviewed-changes-requested` via `gh pr edit --add-label`
3. If `reviewed-changes-requested`, also re-spawn Writer Claude with the review comments as context so it can patch the PR.
4. Logs to `/opt/stockbuddy/logs/cc-reviewer.log`.

**Acceptance criteria**:
- A1: PRs opened by the bot get a review within 5 min.
- A2: At least 80% of synthetic-bug PRs get `reviewed-changes-requested`; at least 80% of synthetic-clean PRs get `reviewed-pass`. (Smoke test set: 5 deliberately-buggy + 5 clean PRs.)
- A3: When Reviewer requests changes, Writer is re-spawned and the PR is updated.
- A4: Service survives a Reviewer crash (systemd Restart=on-failure, dedupe via PR number lock file).
- A5: Reviewer NEVER reviews its own PRs (loop prevention via `author == cc-bot` check).

### 4.2 CI gate (G2)

**Purpose**: Block auto-merge until all required GitHub Actions checks are green.

**Approach**: cc-merger polls `gh pr checks <number>` every 30s for up to 30 min. If any check FAILS, label `ci-failed`, spawn Writer with failure logs as context. If all PASS, proceed to merge.

**Required checks per repo**:
- TTE repo: pytest (if it exists; verify in §6), pre-commit hooks
- Stock Buddy repo: existing GitHub Actions workflows (don't enumerate here — `gh pr checks` will surface them)

**Acceptance criteria**:
- A6: PR with failing check is NOT merged.
- A7: PR with green checks + `reviewed-pass` label gets merged within 60 s of both conditions being true.
- A8: If CI never finishes within 30 min, label `ci-timeout`, ping Telegram, do not merge.

### 4.3 cc-merger + cc-deployer (G3)

**cc-merger purpose**: Atomic auto-merge when (CI green) ∧ (label = `reviewed-pass`).

**Behavior**:
1. Subscribed to GitHub webhook (`check_suite.completed`, `pull_request.labeled`) on port 8767.
2. On event, re-check both conditions via gh CLI.
3. If satisfied: `gh pr merge --squash --delete-branch <number>`.
4. On successful merge, POST to cc-deployer at `127.0.0.1:8768/deploy` with `{repo, commit_sha, pr_number}`.

**cc-deployer purpose**: Pull the merged commit, rebuild image, restart container.

**Per-repo deploy flow**:

- **TTE** (`/opt/stockbuddy/tte` on KVM8):
  1. `cd /opt/stockbuddy/tte && git fetch origin && git checkout main && git pull --ff-only`
  2. Record pre-deploy health baseline (see §4.4)
  3. `docker build -t tte:phase4 .`
  4. `docker stop tte-1 && docker rm tte-1`
  5. `docker run -d --name tte-1 ... tte:phase4 python -m tte.main --maintain-only` (preserving volumes, env)
  6. Wait 30 s, check `docker logs tte-1 --tail 50` for startup errors; if errors, fire watchdog immediately
  7. Hand off to cc-watchdog

- **Stock Buddy** (Vercel-hosted): no-op — Vercel auto-deploys on push to main. cc-deployer just records the deploy event timestamp + commit sha to `/opt/stockbuddy/cc-state/deploys.jsonl`.

**Acceptance criteria**:
- A9: Within 60s of a merge, deploy fires.
- A10: tte-1 is rebuilt + restarted with the new image; `docker inspect tte-1` shows the new image sha.
- A11: Pre-deploy baseline JSON is written to `cc-state/baseline-<deploy_id>.json`.
- A12: SB deploys are no-op'd but recorded.

### 4.4 cc-watchdog (G4)

**Purpose**: 10-min post-deploy health monitoring with auto-revert.

**Health metrics** (TTE-focused; SB watches its own Vercel via Uptime Kuma):

| Metric | Source | Healthy if… |
|--------|--------|-------------|
| Container uptime | `docker inspect tte-1 --format '{{.State.StartedAt}}'` | restarted after deploy AND still running 10 min later |
| Snapshot success rate (last 5 min) | Mongo: `setup_messages` snapshotStatus=completed / total | ≥ 80% of pre-deploy baseline |
| Webhook delivery to SB (last 5 min) | `docker logs tte-1 \| grep "Webhook sent.*200"` count | ≥ 50% of pre-deploy baseline (5-min window can be sparse) |
| change_symbol error rate (last 5 min) | `docker logs tte-1 \| grep "Failed to change symbol"` count | ≤ pre-deploy baseline + 10 |
| Maintenance loop ticks (last 5 min) | `docker logs tte-1 \| grep "restart_inactive_alerts"` count | ≥ 1 |

**Behavior**:
1. Triggered by cc-deployer at `T+0` with `{deploy_id, repo, commit_sha, baseline_path}`.
2. At `T+10min`, collect post-deploy metrics for the same 5-min window.
3. Compare to baseline. If ANY metric fails → revert.
4. **Revert flow**:
   - `gh pr edit <pr_number> --add-label auto-reverted`
   - `cd /opt/stockbuddy/tte && git revert <commit_sha> --no-edit && git push origin main`
   - This triggers the cc-merger → cc-deployer chain again (with revert commit)
   - Post Telegram digest: "Auto-revert fired on deploy {deploy_id}. Reason: {metric}. Reverted to {prev_sha}."
5. On healthy outcome at T+10min, post Telegram digest "Deploy {deploy_id} verified healthy" and clear baseline files older than 7 days.

**Acceptance criteria**:
- A13: Synthetic regression test — manually drop snapshot success rate to 0% within 5 min of deploy → watchdog reverts within 11 min of deploy.
- A14: Synthetic healthy test — clean deploy → watchdog posts "verified healthy" Telegram within 11 min.
- A15: Auto-revert is idempotent — repeated regression on revert commit doesn't infinite-loop (lock file `cc-state/revert-lock-<commit>`).
- A16: Watchdog state persists across systemd restart (recovers in-flight watch from `cc-state/watches/<deploy_id>.json`).

### 4.5 TTE monitoring checks (G5 / Option C)

**Location**: `/opt/stockbuddy/monitoring/checks/tte-*.sh` (new directory + files).

**Cron schedule**: New systemd timer `tte-checks.timer` firing every 5 min.

**Checks**:

| File | What it checks | Threshold | Alert payload |
|------|---------------|-----------|---------------|
| `tte-snapshot-success-rate.sh` | Mongo: % of setup_messages with snapshotStatus=completed (last 1h) | <80% → alert | `{type: "tte.snapshot.low_success", rate: 0.X, window: "1h"}` |
| `tte-snapshot-failed-burst.sh` | Mongo: count snapshotStatus=failed (last 30 min) | >10 → alert | `{type: "tte.snapshot.failed_burst", count: N}` |
| `tte-change-symbol-errors.sh` | `docker logs tte-1 --since 30m \| grep "Failed to change symbol" \| wc -l` | >5 → alert | `{type: "tte.selenium.change_symbol", count: N}` |
| `tte-alert-sidebar-errors.sh` | `docker logs tte-1 --since 30m \| grep -E "alerts sidebar\|tabpanel_log" \| wc -l` | >5 → alert | `{type: "tte.selenium.alert_sidebar", count: N}` |
| `tte-container-restart.sh` | `docker inspect tte-1 --format '{{.RestartCount}}'` delta from last check | Δ > 0 → alert | `{type: "tte.container.unexpected_restart", new_count: N}` |
| `tte-webhook-delivery.sh` | `docker logs tte-1 --since 30m \| grep "Webhook sent.*200" \| wc -l` (only during market hours) | 0 → alert | `{type: "tte.webhook.no_delivery", window: "30m"}` |

Each check is a small bash script that exits 0 if healthy, exits non-zero AND POSTs to cc-trigger if alerting. State (last alert timestamp per check) in `/opt/stockbuddy/cc-state/check-state/`.

**Dedupe**: each check has a `min_interval_between_alerts = 1h` to avoid spam. Re-alert if condition persists past that window.

**Acceptance criteria**:
- A17: Each of the 6 checks runs every 5 min via the timer.
- A18: Inject each failure mode synthetically, confirm cc-trigger receives the alert.
- A19: Dedupe works — same failure for 1h doesn't fire more than 1 alert.
- A20: Healthy state during market hours → no alerts in 24h soak test.

## 6. Open questions before implementation

1. **TTE pytest**: does the repo have CI? `gh pr checks` will tell me; if no CI exists, set up a minimal GitHub Action that runs `pipenv run pytest tests/`.
2. **GitHub webhook delivery to KVM8**: KVM8's public IP must accept webhooks on a chosen port; alternatively, use a polling approach (`gh pr list --label needs-review`). Polling is simpler and avoids opening firewall ports — recommend polling at 30s cadence.
3. **Reviewer Claude OAuth**: cc-reviewer runs as user `stockbuddy`; needs its own `claude` OAuth session. Either share with cc-trigger's existing session or run a separate one.
4. **Health baseline window**: 5 min is short. During quiet markets, snapshot rate may be sparse. Should baseline be 5 min OR fixed thresholds (e.g., snapshot success >= 80% absolute regardless of pre-deploy)? **Recommend hybrid**: use baseline if pre-deploy count is >=10, else absolute threshold.
5. **Auto-revert at night**: should auto-revert be paused during off-market hours (low alert volume = noisy false positives)? **Recommend yes**: 10-min watchdog only enabled if market is open in any region.

These questions don't block writing the spec; they need answers before implementation.
